# Work queues

첫 번째 튜토리얼에서는 명명된 큐에서 메시지를 주고받는 프로그램을 작성했습니다. 이번 튜토리얼에서는 multiple workers에게 시간이 많이 소비되는 배포 작업을 진행할 Work Queue를 만들겠습니다.

Work Queues(Aka: Task Queues)의 주요 개념은 리소스 집약적인 작업을 즉시 수행하고 완료될 때까지 기다려야 하는 것을 방지하는 것입니다. 대신에 우리는 그 일을 나중에 하도록 스케줄을 잡습니다. 태스크를 메시지로 캡슐화하여 대기열에 보냅니다. 백그라운드에서 실행 중인 작업자 프로세스는 태스크를 팝업하고 최종적으로 작업을 실행합니다. 많은 작업자를 실행하면 작업이 공유됩니다.

이 개념은 짧은 HTTP 요청 윈도우에서 복잡한 작업을 처리할 수 없는 웹 응용 프로그램에서 특히 유용합니다.

### 준비

이전 튜토리얼에서 "Hello World!"라는 메시지를 보냈습니다. 이제 복잡한 작업을 나타내는 문자열을 보낼 것입니다. 크기 조정해야 할 이미지나 렌더링해야 할 pdf 파일과 같은 실제 작업이 없기 때문에 `Thread.sleep()`함수를 사용하여 바쁜것처럼 흉내를 내겠습니다. 문자열의 점 수를 복잡성으로 간주할 것입니다. 각각의 점은 1초가 걸리는 작업입니다. 예를 들어 가짜 작업인 `Hello...`는 3초가 걸리는 가짜 작업으로 묘사될 것입니다.

커맨드 라인에서 임의의 메시지를 보낼 수 있도록 이전 튜토리얼의 예제인 Send.java 코드를 약간 수정하겠습니다. 이 프로그램은 work queue에 따라 task를 스케줄링하기 떄문에 이름을 `NewTask.java`로 지정하겠습니다.

```
String message = String.join(" ", argv);

channel.basicPublish("", "hello", null, message.getBytes());
System.out.println(" [x] Sent '" + message + "'");
```

Recv.java 프로그램도 약간 변경해야 합니다. 이 프로그램은 메시지 본문에 있는 모든 점이 1초의 작업인 것 처럼을 묘사해야 합니다. 배달된 메시지를 처리하고 작업을 수행하므로 `Worker.java`라고 하겠습니다.

```
DeliverCallback deliverCallback = (consumerTag, delivery) -> {
  String message = new String(delivery.getBody(), "UTF-8");

  System.out.println(" [x] Received '" + message + "'");
  try {
    doWork(message);
  } finally {
    System.out.println(" [x] Done");
  }
};
boolean autoAck = true; // acknowledgment is covered below
channel.basicConsume(TASK_QUEUE_NAME, autoAck, deliverCallback, consumerTag -> { });
```

실행 시간을 시뮬레이션하는 가짜 작업입니다.

```
private static void doWork(String task) throws InterruptedException {
    for (char ch: task.toCharArray()) {
        if (ch == '.') Thread.sleep(1000);
    }
}
```

작업 디렉토리의 jar파일 및 환경 변수 `CP`를 이용해 이전 튜토리얼과 같은 방법으로 컴파일합니다.

```
javac -cp $CP NewTask.java Worker.java
```

### Round-robin dispatching

작업 대기열을 사용할 때의 장점 중 하나는 작업을 쉽게 병렬화하는 기능입니다. 만약 우리가 밀린 일을 쌓고 있다면, 우리는 더 많은 노동자를 추가하고, 그렇게 하면 쉽게 확장할 수 있습니다.

먼저 두 개의 작업자 인스턴스를 동시에 실행해 보겠습니다. 둘 다 queue에서 메시지를 받겠지만 정확히 어떻게 될까요? 확인해봅시다.

세 개의 콘솔을 열고, 두 개의 worker프로그램을 실행합니다. 이 두 콘솔은 C1과 C2라는 두 consumer가 될 것입니다.

```
# shell 1
java -cp $CP Worker
# => [*] Waiting for messages. To exit press CTRL+C
```

```
# shell 2
java -cp $CP Worker
# => [*] Waiting for messages. To exit press CTRL+C
```

세 번째 콘솔에서는 새로운 task을 게시합니다. consumers를 시작한 후에는 다음과 같은 몇 가지 메시지를 publish할 수 있습니다.

```
# shell 3
java -cp $CP NewTask First message.
# => [x] Sent 'First message.'
java -cp $CP NewTask Second message..
# => [x] Sent 'Second message..'
java -cp $CP NewTask Third message...
# => [x] Sent 'Third message...'
java -cp $CP NewTask Fourth message....
# => [x] Sent 'Fourth message....'
java -cp $CP NewTask Fifth message.....
# => [x] Sent 'Fifth message.....'
```

worker에 어떤 메시지가 도착했는지 확인해봅시다.

```
java -cp $CP Worker
# => [*] Waiting for messages. To exit press CTRL+C
# => [x] Received 'First message.'
# => [x] Received 'Third message...'
# => [x] Received 'Fifth message.....'
```

```
java -cp $CP Worker
# => [*] Waiting for messages. To exit press CTRL+C
# => [x] Received 'Second message..'
# => [x] Received 'Fourth message....'
```

기본적으로 RabbitMQ는 각 메시지를 순서대로 다음 consumer에게 전송합니다. 평균적으로 모든 consumer가 동일한 수의 메시지를 수신합니다. 이러한 메시지 배포 방법을 round-robin이라고 합니다. 이 방식을 세개 이상의 worker로 시도해보세요.

### Message acknowledgment

작업을 수행하는 데 몇 초가 걸릴 수 있습니다. 만약 소비자 중 하나가 테스크를 수행 중 테스크를 마치지 못하고 죽게된다면 어떻게 될까요? RabbitMQ는 메세지를 소비자에게 전달할 때 전달된 메세지를 즉시 삭제 표시합니다. 이 경우, 만약 당신이 소비자를 정지시킨다면, 소비자는 처리중인 메시지를 잃게 될 것입니다. 또한 특정 소비자에게 발송되었지만 아직 처리되지 않은 모든 메시지도 잃게 될 것입니다.

하지만 우리는 어떤 테스크도 잃고 싶지 않습니다. 소비자가 죽으면 다른 소비자에게 그 테스크를 전달해 주었으면 합니다.

메시지가 손실되지 않도록 하기 위해 RabbitMQ은 [message acknowledgments](https://www.rabbitmq.com/confirms.html)(이후 `메시지 수신 확인`으로 표기)을 지원합니다. 소비자가 메시지를 수신하고 나면 ack(acknowledgement)를 RabbitMQ에게 전송하여 메시지가 수신되었고 처리되었으며 RabbitMQ가 이 메세지를 자유롭게 삭제할 수 있음을 알려줍니다.

소비자가 ack 메세지를 RabbitMQ을 전송하지 않고 죽는 경우 (채널이 닫히거나, 커넥션이 닫히거나, TCP 연결이 끊어진 경우) RabbitMQ는 메시지가 완전히 처리되지 않았다고 판단하여 메시지를 다시 큐에 넣습니다. 동시에 온라인에 다른 소비자가 있으면 다른 소비자에게 신속하게 이 메시지를 재전송합니다. 이렇게 하면 소비자들이 가끔 문제가 생기더라도 어떤 메시지도 잃어버리지 않을 수 있습니다.

RabbitMQ는 메시지를 처리하는데 시간 초과가 없습니다. RabbitMQ는 소비자가 중지되면 메시지를 다시 전달합니다. 메시지를 처리하는데 시간이 많이 걸리더라도 괜찮습니다.

[Manual message acknowledgments](https://www.rabbitmq.com/confirms.html)(이후 `수동 메시지 수신 확인`으로 표기)은 기본적으로 켜져 있습니다. 이전 예제에서는 `autoAck=true` 플래그를 통해 명시적으로 해제했습니다. 이제 이 플래그을 `false`으로 설정하고 제대로 메시지 수신 확인을 보내 테스크를 처리해보겠습니다.

```
channel.basicQos(1); // accept only one unack-ed message at a time (see below)

DeliverCallback deliverCallback = (consumerTag, delivery) -> {
  String message = new String(delivery.getBody(), "UTF-8");

  System.out.println(" [x] Received '" + message + "'");
  try {
    doWork(message);
  } finally {
    System.out.println(" [x] Done");
    channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
  }
};
boolean autoAck = false;
channel.basicConsume(TASK_QUEUE_NAME, autoAck, deliverCallback, consumerTag -> { });
```

이 코드를 사용하면 메시지를 처리하는 동안 Ctrl+C를 사용하여 소비자를 중지한다 하더라도 메시지기 손실되지 않습니다. 소비자가 중지되면 메시지 수신 확인이 전달되지 않은 모든 메시지가 다시 전달됩니다.

메시지 수신 확인은 메시지를 전달 받은 채널과 동일한 채널로 전송되어야 합니다. 다른 채널을 사용했음을 확인하려고 하면 채널 레벨 프로토콜 예외가 발생합니다. 더 자세한 내용은 [확인에 대한 문서 가이드](https://www.rabbitmq.com/confirms.html)를 참고하십시오.

> __acknowledgment를 잊은 경우__
> 
> basicAck를 깜빡하는 것은 흔한 실수입니다. 쉽게 할 수 있는 오류이지만 결과는 치명적입니다. 클라이언트가 종료될 때 메시지가 다시 전달됩니다 임의 재배포처럼 보일 수 있음). 그러나 RabbitMQ는 메시지 수신 확인이 되지 않은 메시지를 버리지 않기 떄문에 메모리를 계속 잡아먹게 됩니다.
>
> 이러한 실수를 디버깅하기 위해 rabbitmqctl을 통해 message_unacknowleded 필드를 출력할 수 있습니다.


### Message durability

우리는 소비자가 중지되어도 테스크가 없어지지 않도록 하는 방법에 대해 알아보았습니다. 하지만 여전히 RabbitMQ가 중지되면 테스크가 사라지는 문제가 남아있습니다.

RabbitMQ가 중지되거나 충돌할 때, 사용자의 지시가 없다면 한 큐와 메시지가 모두 사라질 것입니다. 메시지가 손실되지 않도록 하려면 큐와 메시지의 durable(이후 `내구성` 으로 표시)을 설정해야 합니다.

먼저, RabbitMQ가 큐를 잃지 않도록 해야 합니다. 이를 위해서는 큐의 내구성을 선언해야 합니다.

```
boolean durable = true;
channel.queueDeclare("hello", durable, false, false, null);
```

이 커맨드는 정확히 설정 되었지만, 지금 설정에서는 작동하지 않을 것입니다. 이미 내구성 설정을 하지않은 `hello`라는 대기열을 정의했기 때문입니다. RabbitMQ에서는 기존 큐를 파라미터로 재정의할 수 없으며 이를 시도하는 모든 프로그램에 오류를 반환합니다. 그러나 간단한 해결 방법이 있습니다. `task_queue`와 같이 이름이 다른 큐를 선언하는 것입니다.

```
boolean durable = true;
channel.queueDeclare("task_queue", durable, false, false, null);
```

이 `queueDeclare` 변경 사항은 생산자 코드와 소비자 코드 모두에 적용해야 합니다.

이제 `task_queue`큐가 RabbitMQ가 재시작 되더라도 없어지지 않을 것입니다. 이제 `BasicProperties`를 구현한 `MessageProperties`를 `PERSISTENT_TEXT_PLAIN` 값으로 설정하여 메시지를 영구적으로 표시하겠습니다.

```
import com.rabbitmq.client.MessageProperties;

channel.basicPublish("", "task_queue",
            MessageProperties.PERSISTENT_TEXT_PLAIN,
            message.getBytes());
```

> __메시지 지속성에 대한 참고 사항__
>
> 메시지를 영구적으로 표시한다고 해서 메시지가 손실되지 않는 것은 아닙니다. 이 설정이 RabbitMQ가 메시지를 디스크에 저장하도록 하지만 RabbitMQ가 메시지를 수신하고 저장기 바로 전에 짧은 틈이 있습니다. 또한 RabbitMQ가 모든 메시지에 대해 `fsync(2)`를 수행하지는 않습니다. 이 기능은 캐시에 저장되고 디스크에 실제로 기록되지 않을 수도 있습니다. 지속성 보장은 완벽하지 않지만, 샘플로 작성한 테스크 큐에는 충분합니다. 더 강력한 지속성 보장이 필요한 경우 publisher confirms을 사용할 수 있습니다.

### Fair dispatch

여러분은 아마 메시지 전송이 여전히 원하는 대로 정확히 작동하지 않는다는 것을 알아차렸을지도 모릅니다. 예를 들어, 두 소비자가 있는 상황에서, 모든 홀수 메시지는 부하가 크고 짝수 메시지는 부하가 작을 때, 한 소비자자는 계속해서 메세지를 처리해야 하지만 다른 소비자는 거의 메세지를 처리하지 않을 것입니다.  RabbitMQ는 이런 상황을 모르고 여전히 고르게 메시지를 보낼 것입니다.

이러한 문제는 RabbitMQ가 큐에 메시지가 들어올 때 메시지를 보내기만 하기 때문에 발생합니다. 이런 배포 방식은 소비자로부터 전달된 메세지중 메시지 수신 확인이 되지 않은 메시지 수를 확인하지 않고 맹목적으로 모든 n번째 메시지를 n번째 소비자에게 보냅니다.

`prefetchCount = 1` 설정으로 `basicQos`메서드를 사용할 수 있습니다. 이것은 RabbitMQ에게 한 번에 두 개 이상의 메시지를 소비자에게 전달하지 않도록 합니다. 즉, 이전 메시지를 처리하고 승인할 때까지 소비자에게 새로운 메시지를 발송하지 않습니다. 대신 RabbitMQ는 바쁘지 않은 다음 소비자에게 메시지를 발송할 것입니다.

```
int prefetchCount = 1;
channel.basicQos(prefetchCount);
```

> __대기열 크기에 대한 참고 사항입니다.__
>
> 모든 소비자가 바쁠 경우 큐가 가득 찰 수 있습니다. 큐를 확인하고, 더 많은 소비자를 추가하거나, 아니면 다른 전략을 사용해 할 수 도 있습니다.

### __종합해 봅시다.__

`NewTask.java` 클래스의 최종 코드입니다.

```
import com.rabbitmq.client.Channel;
import com.rabbitmq.client.Connection;
import com.rabbitmq.client.ConnectionFactory;
import com.rabbitmq.client.MessageProperties;

public class NewTask {

  private static final String TASK_QUEUE_NAME = "task_queue";

  public static void main(String[] argv) throws Exception {
    ConnectionFactory factory = new ConnectionFactory();
    factory.setHost("localhost");
    try (Connection connection = factory.newConnection();
         Channel channel = connection.createChannel()) {
        channel.queueDeclare(TASK_QUEUE_NAME, true, false, false, null);

        String message = String.join(" ", argv);

        channel.basicPublish("", TASK_QUEUE_NAME,
                MessageProperties.PERSISTENT_TEXT_PLAIN,
                message.getBytes("UTF-8"));
        System.out.println(" [x] Sent '" + message + "'");
    }
  }
}
```

`Worker.java` 클래스의 최종 코드입니다.

```
import com.rabbitmq.client.Channel;
import com.rabbitmq.client.Connection;
import com.rabbitmq.client.ConnectionFactory;
import com.rabbitmq.client.DeliverCallback;

public class Worker {

  private static final String TASK_QUEUE_NAME = "task_queue";

  public static void main(String[] argv) throws Exception {
    ConnectionFactory factory = new ConnectionFactory();
    factory.setHost("localhost");
    final Connection connection = factory.newConnection();
    final Channel channel = connection.createChannel();

    channel.queueDeclare(TASK_QUEUE_NAME, true, false, false, null);
    System.out.println(" [*] Waiting for messages. To exit press CTRL+C");

    channel.basicQos(1);

    DeliverCallback deliverCallback = (consumerTag, delivery) -> {
        String message = new String(delivery.getBody(), "UTF-8");

        System.out.println(" [x] Received '" + message + "'");
        try {
            doWork(message);
        } finally {
            System.out.println(" [x] Done");
            channel.basicAck(delivery.getEnvelope().getDeliveryTag(), false);
        }
    };
    channel.basicConsume(TASK_QUEUE_NAME, false, deliverCallback, consumerTag -> { });
  }

  private static void doWork(String task) {
    for (char ch : task.toCharArray()) {
        if (ch == '.') {
            try {
                Thread.sleep(1000);
            } catch (InterruptedException _ignored) {
                Thread.currentThread().interrupt();
            }
        }
    }
  }
}
```

메시지 수신 확인 및 prefetchCount를 사용하여 테스크 큐를 설정할 수 있었습니다. 내구성 옵션을 사용하면 RabbitMQ를 다시 시작해도 태스크가 유지될 것 입니다.

채널 메소드 및 MessageProperties에 대한 자세한 내용은 온라인에서 JavaDocs를 확인하시기 바랍니다.

이제 튜토리얼 3으로 넘어가서 많은 소비자에게 동일한 메시지를 전달하는 방법을 학습 하겠습니다.
